# 从零开始搭一个纯 Markdown 博客

- 标签：GitHub, Markdown, 写作
- 状态：published

## 为什么我要这样写博客

我想要一个足够轻的写作方案：

- 不依赖框架
- 不折腾构建流程
- 打开仓库就能读
- 以后迁移到别的系统也方便

相比先搭一套完整博客系统，我更想先把内容写起来。纯 Markdown 的好处是简单、稳定，也更容易长期维护。

## 这套目录怎么分工

这个仓库把内容和资源拆得比较直接：

- `posts/` 放正式文章
- `series/` 放系列文章
- `notes/` 放草稿、灵感和临时片段
- `assets/` 放图片和附件
- `templates/` 放固定模板

这样做的好处是写作路径很清楚：先记，再整理，再发布。

## 文件命名为什么用日期加 slug

我给正式文章采用 `YYYY-MM-DD-slug.md` 的命名方式，例如：

`2026-03-22-start-a-pure-markdown-blog.md`

这个规则有几个好处：

- 文件天然按时间排序
- 标题改了，文件名也不一定要跟着改
- 迁移到其他博客工具时更容易兼容
- 一眼就知道文章大概是什么时候写的

## 纯 Markdown 方案适合什么人

如果你也有下面这些想法，这种方式会比较合适：

- 你想先稳定写，而不是先折腾站点
- 你主要在 GitHub 上阅读和管理内容
- 你愿意手动维护索引、归档和标签页
- 你看重内容文件本身的可迁移性

## 现在就可以开始写了

博客的起点不一定是一个完整的网站，也可以只是一个结构清楚、能持续更新的仓库。

先把第一篇写出来，比先挑框架更重要。
